🎓 Computer Security & Forensics — Final Year Project Context

EduGuard MDM demonstrates applied security concepts: access control (app whitelisting), policy enforcement (Android Device Policy Controller), digital forensics (immutable audit logs), cryptographic integrity (AES-256 signed policy bundles), and resilience engineering (offline-first sync). This MVP constitutes a fully operational Proof of Concept.

Process

How EduGuard MDM Works

From policy creation to classroom enforcement — a three-phase workflow designed to operate entirely on a local network with no cloud dependency.

01

Admin Configures Policy

The ICT administrator uses the EduGuard web panel to define per-subject or per-session policies: allowed apps, blocked sites, screen time, and exam lockdown rules. Policies are encrypted and signed before storage.

02

Tablets Sync via Local Network

Student tablets pull policy bundles over local Wi-Fi, USB tethering, or Bluetooth. Each bundle is cryptographically verified on-device. No internet required — the Raspberry Pi server handles all distribution.

03

Policy Enforced in Class

The EduGuard Android agent enforces policies continuously. All actions are logged to a tamper-evident audit trail. Violations trigger real-time alerts to the admin dashboard — critical evidence for forensic review.

Capabilities

Security Features Built for Schools

Every feature maps to a core computer security or digital forensics principle — making EduGuard MDM both a working school tool and a rigorous academic proof of concept.

Offline-First Sync

Policy bundles are cached on-device and applied without network connectivity. Sync occurs opportunistically via local Wi-Fi or USB — ensuring zero disruption in rural schools with unreliable internet.

Resilience

App Whitelisting

Only administrator-approved applications can run during sessions. This implements mandatory access control (MAC) at the Android kernel level, preventing unauthorised software execution — a core security principle.

Access Control

Session Scheduling

Policies activate and deactivate on a timetable — automatically locking Mathematics apps during English class, or enabling exam kiosk mode at the scheduled start time. Time-based access control.

Policy Engine

Forensic Audit Logs

Every policy change, app launch, sync event, and violation is written to a hash-chained, append-only log. Logs are exportable as evidence. Tamper-evident by design — directly applicable to digital forensics methodology.

Digital Forensics

QR Code Enrollment

Secure zero-touch device onboarding via signed QR codes. Each enrollment token is single-use, time-limited, and cryptographically bound to a device profile — preventing unauthorised device registration.

Secure Onboarding

Kiosk / Exam Mode

Single-app lockdown for exams. Disables notifications, prevents screenshots, blocks multitasking, and logs any attempt to break out. Directly addresses academic integrity — a key concern for Ugandan UNEB examinations.

Lockdown Mode
Security Architecture

Designed Around the CIA Triad

EduGuard MDM's architecture is directly grounded in the three foundational pillars of information security — as required for a Computer Security & Forensics final year project.

  • AES-256 Policy Encryption All policy bundles encrypted at rest and in transit. Each device holds a unique symmetric key, rotated on each sync cycle.
  • Digital Signature Verification Policy bundles are signed with the server's RSA-2048 private key. The Android agent verifies the signature before applying any policy.
  • Hash-Chained Audit Logs Each log entry includes the SHA-256 hash of the previous entry, creating a forensically sound, tamper-evident chain of custody.
  • Violation Detection & Alerting Real-time detection of policy violations, with alerts propagated to the admin dashboard over the local network without internet.
  • Role-Based Access Control (RBAC) Three roles with scoped permissions: Super Admin, ICT Teacher, Class Teacher. Principle of least privilege enforced at API level.
